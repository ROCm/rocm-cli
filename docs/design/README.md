# rocm-cli UX Mocks — Three Interaction Surfaces

**Purpose.** Translate the [feature audit](../feature-audit/REPORT.md) and
[improvement strategy](../feature-audit/IMPROVEMENT-STRATEGY.md) into concrete,
reviewable UI mocks. The product is well-built (301 features, 86% implemented,
polish 3.65) but under-discovered (discoverability 2.39, 79 hidden gems). These
mocks attack the gap by reshaping *how users enter and move through the product*,
not by adding capability.

**Grounding.** Every mock is constrained by
[`docs/ux-guidelines.md`](../ux-guidelines.md) (non-technical Windows audience,
deterministic non-LLM setup, approval-by-default, navigable surfaces over typed
strings) and the product intent in
[`ROCm-AI-1-Pager.md`](../../../swproductmgmt/docs/ROCm%20AI/ROCm-AI-1-Pager.md)
(intent → outcome; AAI demos: **serve**, **optimize**, **one-command Strix Halo
setup**, **local image gen**).

> These are **wireframes for a terminal UI**, so the medium is annotated ASCII.
> Box-drawing, glyphs, and color callouts map 1:1 to ratatui primitives the
> codebase already owns (`ui/modal.rs`, `ui/gradient.rs`, `ui/sparkline.rs`,
> `ui/theme.rs`). Nothing here requires a rendering capability we don't have.

## 1. The three surfaces (escalating engagement)

The core idea: **one binary, three front doors**, each meeting a different user
where they are. Today there are effectively two (`rocm <cmd>` and the full dash,
with bare `rocm` falling straight into the dash). We split the front door so the
layman gets a calm, narrow entry and the power user keeps the scriptable one.

| # | Invocation | Surface | Audience | Job |
|---|---|---|---|---|
| **1** | `rocm <command>` (e.g. `rocm doctor`, `rocm serve qwen`) | **Single-command CLI** — plain text in/out, no TUI | Power users, scripts, CI, air-gapped automation | One verb, one answer. Scriptable. |
| **2** | `rocm` (bare, no args) | **Minimal launcher** — single-screen intent menu with a live status strip | First-timers, non-technical Windows/Strix Halo users | "What do you want to do?" → guided outcome, ≤2 keys |
| **3** | `rocm dash` | **Full control room** — landing + outlined tabs | Returning users, operators, observers | Live instrument panel + every guided action in reach |

```
   rocm doctor ───────────────►  [ plain-English status + fix ]      Surface 1
                                  scriptable, no UI

   rocm ──────────────────────►  GPU Radeon 8060S │ 34% │ 18/128 GB     Surface 2
   (bare)                         What would you like to do?
                                    ▸ ◆ Serve   ⚙ Set up   ⚕ Diagnose …
                                      ▣ Open full dashboard →
                                       │
                                       ▼
   rocm dash ─────────────────►  ┌ Home ┐┌Action┐┌Observe┐┌Chat┐    Surface 3
                                  │  bento landing · tiles   │
                                  │  live instruments        │
```

**Why bare `rocm` changes.** Today `rocm` reroutes into the dash
(`ROADMAP.md:26-34`). For the stated audience that is too much surface at once
and violates the spirit of ux-guidelines:15 (a *dedicated* prompt before the main
TUI) and :71 (no composer until a session exists). The minimal launcher becomes
the deterministic first contact; the dash is always one row away
(`Open full dashboard →`) and remains the explicit `rocm dash` target.

See: **[surface-1-cli.md](surface-1-cli.md)** ·
**[surface-2-minimal.md](surface-2-minimal.md)** ·
**[surface-3-dash.md](surface-3-dash.md)** ·
**[responsive.md](responsive.md)** (laptop vs. external-monitor layouts).

> **Rendered previews.** The surface docs below carry annotated ASCII wireframes.
> For pixel-accurate, real-terminal renders of the proposed designs, see
> **[mockups/](mockups/README.md)** — generated by the runnable example
> `crates/rocm-dash-tui/examples/gen_mockups.rs`:
>
> ```bash
> cargo run --release --example gen_mockups -p rocm-dash-tui -- \
>     --output-dir docs/design/mockups
> ```
>
> It paints the redesign to a real ratatui framebuffer (real box-drawing, the
> real `Theme` palette, the real gauge/sparkline widgets) and exports SVGs +
> an `index.html` gallery — **without touching the live `ui::draw` path**.

## 2. The new dash tab model: Home · Action · Observe · Chat

Today's dash has five flat, equally-weighted tabs — Overview, Hardware,
Instances, Bench, Chat (`ui/tabs/mod.rs:20-26`) — rendered as inline highlighted
text, *not* outlined. The audit shows the operational power (serve, install,
engines, doctor, providers) lives in overlays with **no on-screen tab home at
all**, which is a primary discoverability sink.

We reorganize around the verb the user is in:

| New tab | Replaces / absorbs | Optimized for |
|---|---|---|
| **Home** *(default landing)* | *new* | Glance + next action. Bento tiles. |
| **Action** | serve wizard, install/engine/doctor/update/provider overlays | **Doing.** Guided wizards as visible tiles. Rescues the slash-command gems (F193–F208). |
| **Observe** | Overview + Hardware + Instances + Bench | **Watching.** Live instruments, tiles, richer telemetry. |
| **Chat** | Chat (unchanged in spirit) | The agent. `/plan` surfaced as a visible chip (F207). |

This folds 5 flat tabs into 3 intent tabs + a landing, and gives the 79 hidden
gems a *named, outlined home* instead of a memorized keybinding.

### Outlined tabs (the explicit ask)

Tabs render as **folder tabs with box-drawing outlines**. The active tab is drawn
in `accent` (cyan) and its bottom edge *opens into the content panel* below it;
inactive tabs are closed boxes in `muted`, sitting on the panel rim.

```
   ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌────────┐
   │ ● Home   │  │  Action  │  │ Observe  │  │  Chat  │   Esc menu  t theme  ? help
 ┌─┘          └──┴──────────┴──┴──────────┴──┴────────┴───────────────────┐
 │                                                                          │
 │   …content panel — the active tab connects to it…                        │
 │                                                                          │
 └──────────────────────────────────────────────────────────────────────────┘
        ▲ active: cyan outline, bottom open    ▲ inactive: muted closed box
```

> Legend reminder: `● Home` = the active tab carries a leading `●` dot in accent.
> The mocks below show one active tab per screen; the others stay closed boxes.

## 3. Shared design language

### Color (carries meaning — ux-guidelines:66-68; uses the real theme palette)

| Token | default-dark RGB | Meaning |
|---|---|---|
| `accent` cyan | `#00c2de` | the single active focus / selection |
| `ok` green | `#1aa01a` | ready, safe, success |
| `warn` amber | `#f59e0b` | work-in-progress, caution, demo data |
| `err` AMD red | `#ed1c24` | **only** real errors / destructive danger |
| `fg` / `muted` | `#eaebec` / `#b4b9bc` | primary / secondary text |
| `surface` / `surface_2` | `#1d1f21` / `#27272a` | tile fill / chip fill |
| `bg` slate | `#131416` | canvas |

AMD red is reserved strictly for danger — never decoration. A deliberate default
theme is shipped (not auto-dark by accident); 15 themes already exist
(F111/F259) and become reachable from the palette and footer.

### Glyph language (consistent, with an on-screen legend — fixes F217)

```
 ●  active / running        ✓  ready / success        ⏸  paused
 ◌  stopped / idle          ⚠  warning / attention     ↺  rollback / revert
 ▸  selected row / cursor   …  in progress (spinner)   →  leads to next surface
```

Verb icons (stable per action, used in the minimal launcher and Action tiles):

```
 ◆  serve a model     ⚙  set up / install     ⚕  diagnose & fix
 ◷  chat              ⌬  engines              ⇲  update
 ⮌  providers/keys    ▣  open dashboard       ⚡  optimize  (soon)
```

### Live vs. aspirational (honesty)

Mocked features are only drawn as live if they exist in the codebase today. A
capability that's planned but unbuilt renders **dimmed with an amber `soon`
badge** and an explicit "planned" sublabel — never as an actionable row.

- **Optimize a model** is `soon` (planned hyperloom skill — no optimize/tuning
  command exists yet; the bench tab only *displays* external results).
- An earlier draft included **Generate an image**; it was removed because the
  shipped `rocm comfyui` surface is ComfyUI server lifecycle management, not an
  in-dash prompt→image flow.

### Footer = single source of truth for "what can I do here?"

Every screen ends with a context chip row driven by the same `KeyAction` map the
reducer dispatches (F110), so footer and behavior cannot drift. The header chrome
**always** exposes `Esc menu`, `t theme`, and `? help`; the command palette
remains available via `:` for power users (it's just no longer headlined).

```
  ↑↓ move   Enter open   Tab next   t  theme   ?  help   q quit
```

### Global menu (`Esc`)

`Esc` opens a btop-style **main menu** over a grey-dimmed backdrop: a gradient
block "ROCM" logo and a vertical **Options / Help / Quit** list (↓ to cycle).

- **Options** — a tabbed settings panel (General · CPU · GPU · Engines) of
  toggles/cycles (theme, start screen, refresh interval, ask-vs-full-access,
  local-only telemetry, soft bell, reduce-motion, units, default engine…).
- **Help** — a two-column keyboard reference grouped by Navigate / Overlays /
  Actions / Chat / Global.
- **Quit** — confirm-and-exit.

See the rendered scenes in [mockups/](mockups/README.md) (`dash-menu`,
`dash-options`, `dash-help`).

### Honesty rules baked into the mocks

- **Demo data** is labeled with a persistent amber banner, never silent (F151).
- **Empty panes teach the next action** instead of going blank (F114/F126).
- **Unpopulated fields** render `N/A`, never a misleading `0` (F124/F132).
- **Mutating actions** route through the approval modal; default is "ask"
  (F172/F173/F206; ux-guidelines:27).

## 4. How these mocks map to the strategy phases

| Mock element | Strategy item | Audit finding rescued |
|---|---|---|
| Minimal launcher + first-run auto-setup | W1.3 | F104 (onboarding never triggers) |
| Command palette `:` overlay | W1.2 | F263 + F193–F208 slash cluster |
| Action tab as visible tiles | §3.1/§3.2 | 79 hidden gems get a home |
| Observe tab hero instruments | W2.1 | F134 tokens/W, F255/F257 promoted |
| Outlined tabs + focus states | W2.2 | design-quality bar (depth, hierarchy) |
| Demo banner, empty-state cards, N/A | W1.4 | F151, F114, F124, F132 |
| Actionable CLI errors | W3.4 | F002, F005, F085 dead-ends |

These mocks are **Phase-1/Phase-2 oriented**: they surface existing polish
through navigation and visual hierarchy. No new telemetry or engine capability is
assumed.
